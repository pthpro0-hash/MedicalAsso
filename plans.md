# 촉탁의 추천·등록·현황관리 시스템 개발 계획서

## 0. 프로젝트 목적

이 프로젝트는 **지역의사협의회/지역의사회 사무국이 촉탁의사 관련 행정 업무를 표준화하고 자동화하는 내부 업무관리 솔루션**을 개발하는 것이다.

핵심은 의료행위 관리가 아니다.  
이 시스템은 환자 진료, 처방, EMR, 공단 청구를 직접 처리하지 않는다.

본 시스템의 목적은 다음 7가지다.

1. 촉탁의사 후보자 DB 관리
2. 요양원 DB 관리
3. 요양원의 촉탁의 추천 요청 접수
4. 의사회 사무국의 후보 검토 및 복수 추천
5. 추천서/공문 자동 생성
6. 최종 지정 결과 및 계약 기간 관리
7. 민원·재추천·계약만료 이력 관리

---

## 1. 서비스 방향 재검토

### 1.1 이 서비스가 해결할 문제

지역의사회 촉탁의 관련 실무는 다음과 같은 문제가 발생하기 쉽다.

| 문제 | 설명 |
|---|---|
| 엑셀·전화·이메일 중심 관리 | 담당자 변경 시 인수인계가 어렵고 이력 누락 가능 |
| 추천 요청 누락 | 요양원 요청이 여러 경로로 들어오면 처리 상태 파악이 어려움 |
| 촉탁의 후보 검색 어려움 | 의사별 가능 지역, 전문과목, 교육 이수 여부를 한 번에 보기 어려움 |
| 과다 배정 | 특정 촉탁의사에게 시설이 몰릴 수 있음 |
| 계약 만료 누락 | 1년 단위 계약 만료, 재계약, 재추천 시점을 놓칠 수 있음 |
| 민원 이력 분산 | 재추천 사유, 시설 불만, 촉탁의 불만이 기록으로 남지 않음 |
| 행정 증빙 부족 | 언제 요청받고 누구를 추천했는지 근거 관리가 약함 |

### 1.2 하지 말아야 할 것

초기 MVP에서는 아래 기능을 개발하지 않는다.

| 제외 항목 | 제외 이유 |
|---|---|
| 환자별 진료기록 | 의료기관/요양원/촉탁의 영역이며 법적·보안 리스크 큼 |
| 처방 관리 | 의사의 의료행위 영역 |
| EMR 대체 | 진입 장벽이 높고 범위가 지나치게 큼 |
| 국민건강보험공단 청구 연동 | 공단 시스템과의 연동 난이도 및 정책 리스크 존재 |
| 요양원 ERP 대체 | 본 서비스의 고객은 의사회 사무국이며 범위 초과 |
| 의료 판단 AI | 법적 책임 위험이 큼 |
| 자동 배정 확정 | 의사회 담당자의 검토와 최종 판단이 필요 |

### 1.3 반드시 지켜야 할 포지션

> **진료는 의사가, 운영은 요양원이, 추천 행정은 의사회가 한다.  
> 이 시스템은 의사회의 추천 행정을 정확하고 빠르게 처리하는 도구다.**

## 2. 핵심 사용자

### 2.1 의사회 관리자

| 사용자 | 목적 |
|---|---|
| 사무국 직원 | 접수, 후보 검색, 추천서 생성, 지정 결과 등록, 만료 관리 |
| 담당 임원/검토 의사 | 추천 후보 검토, 민원 검토, 재추천 승인 |
| 회장/관리자 | 전체 현황, 미배정 시설, 민원, 만료 예정 건 확인 |

### 2.2 촉탁의사

초기 MVP에서는 직접 로그인하지 않아도 된다.  
의사회 직원이 촉탁의사 정보를 대리 등록한다.

2차 버전에서 촉탁의사가 직접 정보 수정, 교육 이수증 업로드, 담당 시설 확인을 할 수 있다.

### 2.3 요양원

초기 MVP에서는 직접 로그인하지 않아도 된다.  
의사회 직원이 요양원 추천 요청을 대리 등록한다.

2차 버전에서 요양원이 직접 추천 요청, 후보 확인, 지정 통보, 재추천 요청을 할 수 있다.

## 3. MVP 개발 원칙

### 3.1 관리자 우선

처음에는 외부 포털을 만들지 않는다.  
**의사회 사무국 직원이 내부 업무를 처리하는 관리자 시스템**으로 시작한다.

### 3.2 수기 입력을 허용

초기에는 모든 연동을 피한다.

- 지도 API 연동 없음
- KMA 교육센터 연동 없음
- 공단 청구 연동 없음
- 요양원 ERP 연동 없음
- 전자계약 연동 없음

대신 파일 업로드, 체크박스, 날짜 관리, PDF 생성으로 시작한다.

### 3.3 상태 기반 워크플로우

추천 요청은 반드시 상태값을 가진다.

```text
접수 → 검토중 → 후보추천완료 → 요양원선택대기 → 지정완료 → 계약등록완료 → 종결
```

상태 변경 이력은 로그로 남긴다.

### 3.4 검수 통과 후 다음 단계

각 개발 파트는 다음 조건을 만족해야 다음 파트로 넘어간다.

1. 기능 구현 완료
2. UI에서 수동 테스트 가능
3. 필수 데이터 생성/수정/삭제 확인
4. 예외 케이스 확인
5. 자체 점검 체크리스트 통과
6. `dev_notes.md`에 완료 내용 기록

## 6. 권장 기술 기준

기존 프로젝트 환경이 없다면 아래 기준으로 개발한다.

### 프론트엔드

- Next.js 또는 React 기반
- TypeScript 사용
- Tailwind CSS 또는 기존 프로젝트 스타일 시스템 사용
- 관리자 중심의 반응형 웹

### 백엔드

- Next.js API Routes / NestJS / Express 중 기존 환경에 맞춤
- REST API 우선
- 모든 주요 액션은 서버에서 검증

### DB

- PostgreSQL 권장
- 개발 초기에는 SQLite도 가능하나, 운영 전 PostgreSQL 전환 고려

### 파일 저장

- 로컬 개발: `/uploads`
- 운영: S3 호환 스토리지 또는 클라우드 스토리지
- 파일 메타데이터는 DB 저장

### PDF 생성

- HTML 템플릿 → PDF 생성 방식 권장
- 추천서, 지정통보서, 내부 검토서부터 시작
